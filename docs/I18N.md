# Internationalization

i18n status: Applies to public report artifacts; English and Spanish gettext
catalogs are at key and placeholder parity.

## Supported surface and entry point

`receipts run --locale en|es` localizes report headings, approval and provenance
language, comparison/reconciliation labels, chart alternative text, receipt
metadata, and the complete trace view. `src/outcome_receipts/copy.py` is the sole
entry point and loads compiled GNU gettext catalogs from
`src/outcome_receipts/locales/`.

The figure display, SQL, author-supplied title/template/definition/caveat, row
count, slice hash, timestamp, and verification semantics remain byte-identical.
Organizations author those fields in the desired language. Operational CLI
messages remain English and are documented as operator-facing rather than part of
the public report artifact.

## Automated gates

`make i18n` regenerates the POT extraction, compares it with the committed
template, compiles every PO file, rejects fuzzy or untranslated entries, checks
EN/ES keys and placeholders, validates BCP 47 tags, and rejects non-UTF-8 authored
files. Tests run report and grounding paths in both locales. The HTML accessibility
gate validates the root language.

CLDR-dependent number, date, time, currency, address, and plural formatting are
not performed by this project: receipt displays and ISO timestamps are evidence
and intentionally do not change across locale. Babel is the pinned catalog and
locale-validation tool; adding localized semantic formatting re-enters the CLDR
freshness and locale-acceptance surface before merge.

## Translation review policy

Catalog changes move through initial, translated, reviewed, and final states in a
pull request. Machine translation may draft but cannot supply the final civic
copy without human review. Placeholder changes are code changes and must pass the
same gate. No TMS or XLIFF round-trip is used at the current two-locale scale; if
one is introduced, XLIFF 2.2 becomes the interchange format.

RTL QA is N/A today because the shipping locales are `en` and `es`, both LTR, and
the output contains no bidi user-generated text. A first RTL locale must add the
full browser checklist and direction-isolation tests before release.

Review artifacts: `docs/audits/lang-of-parts.md`, `docs/LANGUAGE_POLICY.md`,
`docs/compliance-matrix.md`, and `tests/locale_acceptance/`.
